# 问题排查：KSK 一键启动默认选择账号

- 日期：2026-07-14
- 基线 commit：`76aff16` (`docs(ksk-ide): 📝 更新最终能力与验收说明`)
- 问题描述：KSK IDE 的“账号一键启动”在用户未主动选择账号时自动选中第一个可签发账号，并立即允许签发 KSK。
- 真实失败输入：用户截图 `codex-clipboard-9b3671cc-01e7-495b-8cd5-381c22a0c779.png`
- 涉及区域：KAM 前端 / KSK IDE 账号选择状态

## 排查过程

### 1. KSK IDE 页面状态初始化

文件：`src/components/features/KskIde/index.tsx`

`selectedAccountId` 虽然以空字符串初始化，但账号列表加载后，页面 effect 会查找 `firstEligible` 并自动写入其账号 ID。

### 2. 启动按钮状态

文件：`src/components/features/KskIde/index.tsx`

按钮仅以 `selectedAccount` 是否存在作为主要启用条件。自动写入第一个账号后，选择占位符不会显示，按钮也立即可用。

## 发现

- 🐛 [P2] 页面进入时隐式选择第一个可签发账号，用户无需主动确认账号即可签发 KSK — 已修复 `7f0b83e`

## 根因分析

账号列表同步 effect 同时承担“清理失效选择”和“自动填充首个账号”两个职责。后者破坏了显式选择要求，并使下拉框的“选择签发账号”占位状态无法出现。

## 修复与验证

- 空的 `selectedAccountId` 现在保持为空，不再自动写入第一个可签发账号。
- 已选账号消失或失去签发资格时仍会清空选择。
- 回归测试先在旧逻辑上复现 `1 pass / 1 fail`，修复后定向测试 `2/2` 通过。
- 完整前端测试 `33/33`、TypeScript 检查和 Vite 1909 模块 production build 通过。
- 从干净 worktree 的 `7f0b83e` 构建 MSI；MSI 内嵌 EXE 与 Program Files 安装 EXE SHA256 均为 `F2D00C4D2F9D323CC04EB09D1EF0444318017A8AE04518CCE4384FADC5EB340C`。
- 真实安装版初始显示“选择签发账号”且签发按钮禁用；主动选择账号后按钮启用，全程未触发签发。
- 退出后 KAM/Kiro 进程、隔离目录和 endpoint 残留均为 0，正式 token/settings 恢复到启动前 SHA256。

## 结论

修复与安装版验收完成：KSK 一键启动现在必须先由用户显式选择账号。
