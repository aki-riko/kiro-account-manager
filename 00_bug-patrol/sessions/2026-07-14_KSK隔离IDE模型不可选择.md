# 问题排查：KSK 隔离 IDE 模型不可选择

- 日期：2026-07-14
- 基线 commit：`d20497e docs(ksk-ide): 记录共享数据边界与验证结果`
- 问题描述：用户使用安装版 KAM 启动真实 KSK 隔离 Kiro 后，模型下拉框不可选择，并出现本地代理拒绝未授权操作的 JSON 反序列化错误。
- 涉及区域：R2 API 代理与请求转换（KAM `ksk_ide` 原生代理）

## 真实复现证据

- 当前真实会话日志 `Kiro Logs.log` 在 04:29:52 记录 `Error getting available models`。
- 同一时间窗口的 `q-client.log` 记录 `GetUsageLimitsCommand` 被拒；IDE 提示收到纯文本 `KSK 本地代理拒绝未授权操作`，随后 JSON 解析失败。
- 复现结束后，Kiro 进程和隔离目录均为 0；正式 token 与 settings 的 SHA256 均恢复到启动前值。

## 排查过程

### 1. 本地代理白名单

文件：`src-tauri/src/ksk_ide/config.rs`

当前 `is_allowed_operation` 只允许 Runtime `GenerateAssistantResponse`，对 Generic 和 Management 服务一律拒绝。因此 CPS 模型列表必然在本机返回 403。

### 2. 拒绝响应

文件：`src-tauri/src/ksk_ide/proxy.rs`

拒绝分支返回无 JSON Content-Type 的纯文本。官方 AWS SDK 按 JSON 协议解析该响应，产生 `Unexpected token 'K'`，并将可用模型降级为空列表。

### 3. 官方 Kiro 0.12.333 操作证据

文件：`resources/app/extensions/kiro.kiro-agent/dist/extension.js`

首次按 operation schema 读取曾误判为 REST GET；真实失败反证后继续沿实际协议序列化链逆向，最终确认模型列表使用：

- 客户端：`KiroControlPlaneBearerClient`
- 命令：`ListAvailableModelsCommand`
- 协议：`AwsJson1_0Protocol` / `RpcProtocol`
- 方法：`POST`
- 路径：`/`
- 目标：`KiroControlPlaneBearerService.ListAvailableModels`
- 服务：`https://management.{region}.kiro.dev`
- JSON body：`origin`，可选 `profileArn`、`nextToken`、`maxResults`、`modelProvider`
- KSK 认证：`Authorization: Bearer <KSK>` 与 `TokenType: API_KEY`

`RpcProtocol.serializeRequest()` 固定创建根路径、空 query、JSON body 和 POST 方法；operation schema 中的 GET/path trait 不参与当前客户端实际出站。

## 发现

- 🐛 [P1] Management 模型列表 AWS JSON RPC 缺少最小白名单，导致模型列表为空 — 已修复 `131fa6f`
- 🐛 [P1] 隔离占位 `profileArn` 会进入模型 RPC JSON body，KSK 请求必须在代理中删除 — 已修复 `131fa6f`
- ⚠️ [P2] 未授权操作返回纯文本，AWS JSON 客户端会产生误导性的反序列化错误 — 已修复 `9104c2e`

## 根因分析

首版代理按核心聊天最小范围只放行 `GenerateAssistantResponse`，但官方模型选择初始化依赖 CPS `ListAvailableModels`。第一轮修复错误地相信 operation schema 的 GET 路径；实际 `AwsJson1_0Protocol` 继承的 RPC 序列化器固定发送 `POST /`，所以仍未命中白名单。官方 SDK 收到 403 后回退为空模型列表，最终禁用 Select Model。

## 修复结果

1. 仅允许 Management `POST /` 且 `x-amz-target` 精确等于 `KiroControlPlaneBearerService.ListAvailableModels`。
2. 只从模型 RPC 顶层 JSON body 删除占位 `profileArn`，保留 `origin` 和分页字段。
3. Generic、额度、MCP 和其他管理操作继续默认拒绝。
4. 本地拒绝响应改为 `application/x-amz-json-1.0` 结构化 JSON。
5. 新增真实 RPC 方法、路径、target、body 改写和 JSON 响应的端到端 mock 测试。

## 验证

- 真实 KSK management 探针：HTTP 200，15 个模型，默认模型 `auto`。
- 真实 KSK runtime 核心聊天：HTTP 200，`application/vnd.amazon.eventstream`，7 帧、1007 字节完整解析，助手精确返回非敏感测试标记。
- 真实会话停止后：正式 settings/token SHA256 恢复，Kiro/KAM 进程、隔离目录和 endpoint 残留均为 0。
- 最终完整测试：Rust 261/261、Bun 29/29、TypeScript、Vite 1908 模块均通过。
- 最终安装版假 KSK lifecycle：19.75 秒通过。
- 增强后的“最终代理代码 + 安装版 Kiro”模型握手 lifecycle：21.45 秒通过；测试只在代理已取得官方上游 HTTP 响应后记录事件，坐实官方 Kiro 的 `Management/ListAvailableModels` 请求穿过最终白名单。
- 增强 lifecycle 前后正式 settings/token SHA256 完全一致，Kiro/KAM 进程、代理端口和隔离目录残留均为 0。
- 最终 MSI 解包载荷与 Program Files 安装 EXE 的 SHA256 完全一致。

## 结论

根因和核心聊天修复链已由真实失败日志、两轮字节级逆向、真实 KSK management/runtime 请求、自动化测试、最终安装包载荷校验及“最终代理代码 + 安装版 Kiro”真实模型握手共同验证。协议与核心功能证据已闭环；最终安装版 KAM 仍需用户下一次在 UI 重新输入 KSK，完成模型下拉和同一路径聊天的人工验收。KSK 按设计未被持久化，无法也不应在无人值守重启后自动重放。
